# snes-terminal-bridge

Translates keyboard input into SNES controller button sequences and injects them into a running SNES emulator. The goal is to allow typing ASCII text in a terminal, with each character being converted into the corresponding sequence of SNES button presses — the foundation for running terminal applications on a Super Nintendo.

---

## How it works

The bridge applies two mappings in sequence:

```
keystroke
  → mappings.yaml          (ASCII character → SNES button combination)
  → keyboard_mappings.yaml (SNES button → emulator keyboard key)
  → xdotool XTest injection
```

**Example:** typing `H`

1. `mappings.yaml`: `H` → `[A, L]`
2. `keyboard_mappings.yaml`: `A` → `x`, `L` → `d`
3. The bridge presses `x + d` simultaneously via xdotool XTest for 80 ms

`xdotool` XTest injects events at the X server level. bSNES+ polls the X11 keyboard state directly rather than relying on X11 focus, so **no focus switching is performed** — the bridge terminal keeps focus throughout and the user can keep typing immediately.

### Pipeline

```
Thread 1: curses raw keyboard read  →  queue.Queue
Thread 2: dequeue  →  mapper  →  TUI update  →  KeyboardInjector.press_combo()
```

The TUI shows three columns per keypress:

```
snes-terminal-bridge  |  Ctrl+C to quit
  'H'          →  [A, L]               →  x+d
  'e'          →  [A, Select, Right]   →  x+u+Right
  'l'          →  [B, Select, Down]    →  z+u+Down
```

## SNES ROM

The `snes/` subdirectory contains the SNES-side application that runs inside the emulator.

### Current state: interactive input line ✅

The ROM receives joypad combos from the bridge, looks up the corresponding ASCII character, and displays it on screen. Characters appear left-to-right on the top row, one 16×16 tile per character, up to 16 characters.

**Protocol:**
- Each character is encoded as a unique joypad bitmask (combination of SNES buttons held simultaneously for 80 ms)
- The ROM debounces: the bitmask must be stable for ≥ 2 consecutive VBlanks (~33 ms) before triggering
- Same combo is not re-triggered until all buttons are released (no key repeat)
- On match, the tile is written to VRAM at the next VBlank; cursor advances to the next column

`gen_keymap.py` generates `assets/keymap.inc` — a lookup table mapping each joypad bitmask to a font tile number, compiled from `config/mappings.yaml`.

### How the display works

- **BG mode:** SNES BG Mode 0, BG1 only, 16×16 tiles, 2bpp (4 colours)
- **Palette:** colour 0 = black ($0000), colour 1 = white ($7FFF)
- **VRAM layout:**
  - `$0000–$07FF` — BG1 tilemap (32×32 entries × 2 bytes)
  - `$1000–$1BFF` — Font tiles (384 × 8×8 subtiles × 16 bytes)
- **Font generation:** `tools/gen_font.py` uses Pillow to render a monospace TTF at 13 px into 16×16 cells, splits each into four 8×8 subtiles (TL/TR/BL/BR), converts to SNES 2bpp format
- **Data transfer:** Two DMA transfers on reset — tilemap to VRAM $0000, font tiles to VRAM $1000

### Building the ROM

Prerequisites: `sudo apt install cc65` (provides `ca65` + `ld65`)

```bash
cd snes
make          # generates font + keymap, assembles + links → build/terminal.sfc
make run      # build and open in bsnes
make clean    # remove build/
```

The output `build/terminal.sfc` is a LoROM SNES ROM, exactly 32768 bytes.

## Emulator configuration

The key names in `config/keyboard_mappings.yaml` must match the keyboard bindings configured in your emulator (Settings → Input → Port 1 in bSNES+).

**Example (bSNES+ with customised Select binding):**

| SNES Button | Keyboard key | Entry in keyboard_mappings.yaml |
|---|---|---|
| Up | Up arrow | `Up: Up` |
| Down | Down arrow | `Down: Down` |
| Left | Left arrow | `Left: Left` |
| Right | Right arrow | `Right: Right` |
| A | x | `A: x` |
| B | z | `B: z` |
| X | s | `X: s` |
| Y | a | `Y: a` |
| L | d | `L: d` |
| R | c | `R: c` |
| Start | Return | `Start: Return` |
| Select | u | `Select: u` |

Key names follow xdotool / X11 syntax (`Return`, `space`, `Up`, `numbersign`, etc.).

## Usage

```bash
# 1. Start bSNES+ and load the terminal ROM (build/terminal.sfc)
# 2. Activate venv
source .venv/bin/activate
# 3. Run the bridge
python -m snes_terminal_bridge
# Optional: override the emulator window title pattern
python -m snes_terminal_bridge --target bsnes
```

Press **Ctrl+C** to quit.

## Configuration

### config/mappings.yaml

Maps ASCII characters (and special keys) to SNES button combinations.

```yaml
settings:
  hold_ms: 80          # How long buttons are held (ms). Minimum ~17 ms (1 SNES frame).
  release_gap_ms: 20   # Pause between combos for clean rising edges.

mappings:
  "A": [A]
  "a": [A, Select]
  " ": [Up, Left, A]
  "KEY_UP": [Up]       # Special keys use KEY_* names
```

Valid button names: `A`, `B`, `X`, `Y`, `L`, `R`, `Start`, `Select`, `Up`, `Down`, `Left`, `Right`

### config/keyboard_mappings.yaml

Maps SNES button names to xdotool key names injected into the emulator.

```yaml
window: "bsnes"   # Substring of the emulator window title

buttons:
  A: x
  Select: u
  # ...
```

## Tested scenarios

### Scenario 1: WSL2 + WSLg + bSNES+ on Windows 11

**Status:** Working

**System:**
- Windows 11 with WSL2 (Ubuntu 24.04), WSLg enabled
- bSNES-plus v05 installed inside WSL
- Python 3.12

**Steps:**

```bash
sudo apt install xdotool bsnes-plus
git clone <repo-url> && cd snes-terminal-bridge
python3 -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt

# Build and load the ROM in bSNES+
cd snes && make && bsnes build/terminal.sfc &
cd ..

# Run the bridge
python -m snes_terminal_bridge
```

**Why no focus switching is needed:**

bSNES+ polls the X11 keyboard state via its ruby input backend on every frame, independently of which window has focus. XTest events (injected via `xdotool keydown/keyup`) update the X server's key state immediately and are picked up by bSNES+ on its next poll — even though the bridge terminal keeps focus.

**Dead ends investigated:**

| Approach | Result |
|---|---|
| `xdotool key --window <wid>` | Uses `XSendEvent` → events marked synthetic → bSNES+ ignores them |
| `xdotool windowfocus` | Uses `XSetInputFocus` → silently ignored under WSLg/XWayland |
| `xdotool windowactivate` | `_NET_ACTIVE_WINDOW` fails under WSLg (no Linux WM) |
| `WScript.Shell.AppActivate("bsnes")` | Returns `False` — window title starts with game name |
| `SetForegroundWindow` via PowerShell | Focus switches to bSNES+ but permission to restore focus to terminal is lost immediately after |
| Virtual gamepad via uinput | bSNES+ filters all `/devices/virtual/` devices; confirmed with `lsof /dev/input/js0` |
